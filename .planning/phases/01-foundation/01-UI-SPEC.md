## Spacing Scale

Declared values (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge internal padding, inline separators |
| sm | 8px | Button padding (compact), input label gap, list item internal gap |
| md | 16px | Default element spacing, card padding, form field gap |
| lg | 24px | Section padding, card-to-card gap, sidebar padding |
| xl | 32px | Page content padding, modal padding |
| 2xl | 48px | Major section breaks, empty state vertical padding |
| 3xl | 64px | Page-level vertical spacing, login card top offset |

Exceptions:
- Touch targets for icon-only buttons (sidebar collapse toggle, row action menus): minimum 40px x 40px hit area
- Status badge height: 20px (not on scale — fixed pill size for visual consistency in table rows)

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 (regular) | 1.5 |
| Label | 12px | 400 (regular) | 1.4 |
| Heading | 20px | 600 (semibold) | 1.2 |
| Display | 28px | 600 (semibold) | 1.1 |

**Notes:**
- Body (14px / 400 / 1.5): table cell content, form input values, description text, sidebar nav labels
- Label (12px / 400 / 1.4): table column headers, form field labels, badge text, status pills, metadata (timestamp, duration)
- Heading (20px / 600 / 1.2): page titles (`<h1>` on each route), section headings, modal titles
- Display (28px / 600 / 1.1): reserved for login page product name only
- Two weights only: 400 (regular) and 600 (semibold). Weight 500 (medium) is not used anywhere in this phase.
- No font sizes outside this set. Do not introduce 16px, 18px, 24px, or any other value.

## Copywriting Contract

### Global Tone
- Direct, technical, operations-focused. Not consumer-friendly marketing copy.
- The user is an agency admin who understands the platform deeply.
- Avoid emojis in UI copy. Use them only in inline tool status badges (reserved for Phase 3).
- Error messages: describe the problem first, then the resolution path.
- CTA verbs: precise imperative verbs (Create, Save, Deactivate, Add, Remove). Never "Submit."
- Cancel buttons must be context-specific — the bare label "Cancel" is prohibited. Use the form "Keep [thing]" or "Back to [place]" or "Discard Changes" depending on context.

The admin layout uses shadcn/ui `<SidebarProvider>` with a fixed-width sidebar (240px expanded, 48px collapsed icon-only).

| Icon | Label | Route | Active in Phase 1 |
|------|-------|-------|-------------------|
| `Building2` | Organizations | /dashboard/organizations | Yes |
| `Bot` | Assistants | /dashboard/assistants | Yes (scoped to org, or global list) |

| `BookOpen` | Knowledge Base | /dashboard/knowledge | No — disabled |
| `Phone` | Campaigns | /dashboard/outbound | No — disabled |

**Disabled nav items:** Render as `<SidebarMenuButton disabled>` with reduced opacity (opacity-40). No tooltip required in Phase 1. These communicate future capability without implying the features are broken.

#### Sidebar Header
- Product name: "VoiceOps" in 14px semibold

## Page-by-Page Interaction Contracts

### Surface 1: Login Page (/login)

**Layout:** Full-page centered. Background: `background` (#0a0a0a). Card: centered, max-width 400px, `card` background, rounded-lg, p-8.

**Primary focal point:** Sign In button (accent color, full width).

**Load state:** No skeleton — page is static until form submission.

**Interaction flow:**
1. Page renders with email field focused (autofocus)
2. User types email + password
3. Submit: button enters loading state ("Signing in..." + spinner), form fields disabled
4. Success: redirect to /dashboard/organizations (the default landing page)
5. Error: button returns to default state, error message appears above the submit button (not inline with fields)
6. AUTH-04 enforcement: any direct navigation to /dashboard/* while unauthenticated redirects here via middleware; after login, redirect back to the originally requested URL

**Validation:**
- Email: required, valid email format (Zod `z.string().email()`)
- Password: required, min 8 characters
- Validation fires on submit, not on blur

**Responsive:** Card is full-width on mobile (no max-width constraint below 480px), 400px max on desktop.

**No "Forgot password" link in Phase 1** — out of scope per requirements.

---

### Surface 2: Admin Layout Shell

**Persistent across all /dashboard/* routes.**

**Initial load:** Sidebar nav renders immediately from server (no skeleton). Page content area shows a loading skeleton while the page's data fetches.

**Sidebar collapse:**
- Desktop: sidebar starts expanded (240px). User can collapse to icon-only (48px) via `<SidebarTrigger>`. State persists in localStorage via shadcn SidebarProvider.
- Mobile: sidebar is a Sheet overlay, triggered by hamburger icon in header bar.

**Sign Out flow:**
1. User clicks user area in sidebar footer
2. DropdownMenu opens with "Sign Out" item
3. On click: `supabase.auth.signOut()` called, redirect to /login
4. No confirmation dialog — sign out is not destructive

**AUTH-03:** Sign Out accessible from any dashboard route via the sidebar footer user menu.

---

### Surface 3: Organizations List Page (/dashboard/organizations)

**Data:** Fetched server-side. Table renders on server, hydrated on client for actions.

**Primary focal point:** "Create Organization" button (top-right of page header, accent color).

**Table columns:** Name | Status | Created | (Actions)
- Name: text, clickable — navigates to /dashboard/organizations/[id]
- Status: `<Badge>` with status color (see Color section)
- Created: formatted date (e.g. "Mar 15, 2026") using date-fns `format(date, 'MMM d, yyyy')`
- Actions: `<DropdownMenu>` with "Edit Organization" and "Deactivate" / "Reactivate" based on current status

**Create flow:**
1. "Create Organization" button opens a `<Sheet>` sliding in from the right
2. Sheet contains the Organization form (name + status fields)
3. On save: sheet closes, table refetches, success toast appears
4. On error: toast with error, sheet remains open

**Edit flow:**
1. "Edit Organization" in row action dropdown navigates to /dashboard/organizations/[id] (dedicated page, not inline)
   — OR — opens the same Sheet in edit mode (implementation choice, but Sheet pattern is preferred for speed)
2. If Sheet: pre-populated with current values

**Deactivate flow:**
1. "Deactivate" in row action dropdown
2. `<AlertDialog>` opens with confirmation copy (see Copywriting section)
3. On confirm: optimistic UI update (badge changes immediately), API call, error toast on failure with revert
4. On "Keep Organization": dialog closes, no change

**Loading state:** Skeleton — 5 skeleton rows matching table column structure, each row 48px tall with `<Skeleton>` blocks.

**Empty state:** Centered in the table area. Icon: `Building2` (lucide, 48px, muted foreground). Heading. Body. "Create Organization" button.

**Sorting:** Name column sortable (client-side, TanStack Table). No pagination in Phase 1 — full list (admin manages a small number of orgs).

---

### Surface 4: Organization Detail / Edit Form (/dashboard/organizations/[id])

**Layout:** Single-column form, max-width 560px, centered in content area. Page heading above form card.

**Primary focal point:** "Save Changes" button (edit mode) / "Create Organization" button (create mode), both accent color.

**Fields:**
1. Organization Name — `<Input>`, required
2. Status — `<Select>` with options: Active, Inactive, Deactivated

**Load state:** Form fields show `<Skeleton>` blocks while org data fetches server-side.

**Save flow:**
1. User edits fields
2. "Save Changes" pressed: button loading state, form fields disabled
3. Success: toast "Organization saved." Button returns to default.
4. Error: toast "Failed to save organization. Try again." Form re-enabled.

**Deactivate action:**
- Shown as a separate destructive `<Button variant="destructive">` below the form, with a visual separator above it
- Triggers `<AlertDialog>` (not inline confirm)
- Only shown when current status is NOT already "Deactivated"
- "Reactivate" button (variant="outline") shown when status IS "Deactivated"

**Navigation:** "Discard Changes" button (edit mode) or "Back to Organizations" button (create mode) navigates back to /dashboard/organizations (or closes Sheet if using Sheet pattern). Both use `variant="outline"`.

**Responsive:** Full-width form on mobile.

---

### Surface 5: Assistant Mappings Page (/dashboard/assistants or /dashboard/organizations/[id]/assistants)

**Scope decision:** Implement as /dashboard/assistants (global list scoped to admin's org via RLS). Per-org drill-down is accessible from the organization detail page as a tab or linked section.

**Primary focal point:** "Add Assistant" button (top-right of page header, accent color).

**Table columns:** Label | Vapi Assistant ID | Status | Added | (Actions)
- Label: text (friendly name)
- Vapi Assistant ID: monospace text, truncated to ~20 chars with full value on hover (`title` attribute or `<Tooltip>`)
- Status: `<Switch>` inline (active/inactive toggle) — optimistic update
- Added: formatted date
- Actions: `<DropdownMenu>` with "Edit Mapping" and "Remove Mapping"

**Add assistant flow:**
1. "Add Assistant" button opens `<Dialog>` (small form — 2 fields)
2. Form: Label (optional) + Vapi Assistant ID (required)
3. On save: dialog closes, table refetches, success toast
4. On duplicate ID error: toast "This assistant ID is already mapped to an organization."

**Toggle active/inactive:**
1. User clicks `<Switch>` in the Status column
2. Optimistic update: switch flips immediately
3. API PATCH call in background
4. On error: switch reverts, error toast appears

**Remove flow:**
1. "Remove Mapping" in row action dropdown
2. `<AlertDialog>` opens (see Copywriting)
3. On confirm: row removed optimistically, API DELETE call, error toast with revert on failure
4. On "Keep Mapping": dialog closes, no change

**Vapi ID field display:** Use `font-mono text-xs` for the assistant ID value in the table cell to visually distinguish it from regular text.

**Loading state:** Skeleton rows.

**Empty state:** Icon `Bot` (lucide, 48px, muted foreground). Heading + body + "Add Assistant" CTA.

## Loading, Error, and Empty States — Summary

| Surface | Loading | Error | Empty |
|---------|---------|-------|-------|
| Login | Button spinner on submit | Inline error above submit button | N/A |
| Org list | 5 skeleton table rows | Toast + table stays empty | Centered empty state with CTA |
| Org form | Skeleton fields | Toast (non-blocking) | N/A |
| Assistant mappings | 5 skeleton table rows | Toast + optimistic revert where applicable | Centered empty state with CTA |

**Global error boundary:** Root layout wraps dashboard in a Next.js `error.tsx` boundary. Error page shows: "Something went wrong." + "Go to Organizations" button. Does not expose stack traces.

**Network/API failure pattern:** All mutations use try/catch. On failure: sonner toast (destructive style, 5s auto-dismiss). Never block the UI with a modal for non-destructive failures.

## Responsive Behavior

| Breakpoint | Layout Change |
|------------|---------------|
| < 768px (mobile) | Sidebar collapses to Sheet overlay. Tables scroll horizontally. Forms go full-width. Login card goes full-width. |
| 768px–1024px (tablet) | Sidebar shows icon-only (48px). Content area expands. |
| > 1024px (desktop) | Sidebar fully expanded (240px). Max-width constraints on forms (560px). |

**Table responsiveness:** On mobile, the organizations table and assistant mappings table scroll horizontally. Do not stack columns vertically — preserve the tabular structure for data density. Add a horizontal scroll indicator (shadow gradient on right edge).

**Priority on mobile:** Name + Status columns always visible. Created and Actions columns may be cut off (user scrolls right).
